Explanation

@ starts read name, which is parsed until the first white space (after 1338)

header format:
@ <instrument-name>:<runID>:<flowcell ID>:<lane-number>:<tile-number>:<x-pos>: <y-pos> <read number>:<is filtered>:<control number>:<barcode sequence>

for is-filtered: Y – fails (bad); N – passes (good)

our example for read 1: 
@ <M01218>:<95>:<000000000-DBCP2>:<1>:<1101>:<16086>:<1338> <1>:<N>:<0>:<ACTGAGCG+GCGTAAGA>

...note the spaces between between ypos and readnumber (1 or 2 for paired end, allows reads to be matched)
Also notice control number=0, meaning no control bits are on

Second line is sequence bases

Third line starts with "+" and sometimes repeats the read name (often omitted)

Fourth line contains base qualities (BQ +33 = shown ASCII value)

for more information, see: http://gensoft.pasteur.fr/docs/casava/1.8.2/CASAVA1_8_Changes.pdf
